# 轻量网盘资源同步观影房间方案

## 项目定位

本项目定位为一个轻量级同步观影房间系统，面向已经通过 AList 或 OpenList 挂载网盘资源的用户。

系统只负责“选片、房间、视频播放同步或房主推流、房间语音”四个核心能力：

- 选片：从 AList/OpenList 暴露的网盘目录中选择一个视频资源。
- 房间：创建观影房间，其他成员通过房间码或邀请链接加入。
- 播放同步：房间成员打开同一个视频，播放、暂停、拖动进度等状态保持同步。
- 房主推流：房主从本机选择视频，本地播放后通过 WebRTC P2P mesh 推给观众。
- 语音：房间成员可以开启麦克风进行实时语音交流。

系统不负责网盘挂载、账号托管、媒体库管理、视频转码和资源分发。AList/OpenList 是直链同步模式的资源入口；房主推流模式下，本地视频只存在于房主浏览器中，不上传后端。

## 观影模式

v1 支持两种观影模式，创建房间时选择，房间内不切换：

- 直链同步模式：所有成员各自播放同一个 AList/OpenList 视频直链，服务端维护播放状态并同步。
- 房主推流模式：房主选择本地视频文件，本地播放并通过 WebRTC P2P mesh 推流给观众，后端只转发信令。

为避免混淆，文档中的“P2P”只描述 WebRTC 连接拓扑；产品能力称为“房主推流模式”。

## 功能边界

### 需要实现的能力

- 连接一个已部署的 AList/OpenList 服务。
- 浏览网盘目录并选择视频文件。
- 创建房间并生成房间码或邀请链接。
- 创建房间时选择直链同步模式或房主推流模式。
- 加入房间后加载房主选择的视频。
- 同步播放、暂停、进度跳转、播放倍速等基础播放状态。
- 房主推流模式下，房主选择本地视频并推流，观众接收房主媒体流。
- 房主推流模式下，观众控制操作作为请求转发给房主端并默认自动执行。
- 新成员加入时自动同步当前视频和播放位置。
- 房间内支持实时语音。
- 房间无人后自动释放房间状态。

### 默认限制

- 默认面向小房间观影，建议 3 到 8 人以内。
- 默认只有房主拥有播放控制权，避免多人同时操作导致状态冲突。
- 默认不持久化房间历史，房间状态保存在内存中。
- 默认不代理视频流，优先让浏览器直接播放 AList/OpenList 返回的视频直链。
- 房主推流模式下不上传本地视频，不做服务端录制、转码或媒体中转。

## 总体架构

系统采用轻量前后端分离架构：

- 前端负责目录浏览、房间界面、视频播放器、播放事件采集、语音采集和 WebRTC 连接。
- 后端负责 AList/OpenList API 对接、房间状态维护、WebSocket 广播和 WebRTC 信令转发。
- AList/OpenList 负责网盘挂载、文件列表、鉴权和视频资源链接。

推荐架构链路：

1. 用户进入前端页面。
2. 前端请求后端获取 AList/OpenList 文件列表。
3. 用户选择视频后创建或更新房间当前视频。
4. 后端通过 WebSocket 将视频和播放状态广播给房间成员。
5. 各成员浏览器使用 HTML5 video 播放同一个视频 URL。
6. 语音通过 WebRTC 建立浏览器之间的音频连接，后端只负责信令转发。

房主推流模式链路：

1. 房主创建房主推流房间。
2. 房主从本机选择视频文件，浏览器用临时对象 URL 加载到 HTML5 video。
3. 房主点击开始推流，前端通过 `HTMLMediaElement.captureStream()` 采集播放器媒体流。
4. 后端通过 WebSocket 转发房主与观众之间的 WebRTC offer、answer 和 ICE candidate。
5. 观众浏览器接收远端 MediaStream 并播放。
6. 观众播放控制会作为控制请求发送给后端，再转发给房主端自动执行。

## 核心模块

### 选片模块

选片模块负责展示 AList/OpenList 中的视频资源。

核心能力：

- 配置 AList/OpenList 服务地址。
- 获取指定路径下的文件和目录。
- 支持进入子目录。
- 过滤常见视频文件类型，例如 mp4、webm、m3u8、mov。
- 选择视频后获取可播放地址。

选片模块不需要实现复杂媒体库能力，不做海报、演员、简介、刮削和收藏。

### 房间模块

房间模块负责管理观影会话。

核心能力：

- 创建房间。
- 加入房间。
- 离开房间。
- 标记房主。
- 维护成员列表。
- 维护当前视频信息。
- 维护当前播放状态。
- 维护观影模式和房主推流状态。
- 房间无人后自动清理。

房间可以使用短房间码，例如 6 位字母数字组合，也可以同时生成完整邀请链接。

### 播放器模块

播放器模块基于浏览器原生 HTML5 video。

核心能力：

- 加载当前房间的视频 URL。
- 监听播放、暂停、拖动进度、倍速变化等事件。
- 根据服务端广播更新本地播放器状态。
- 对播放时间漂移进行校准。
- 对不可播放格式给出明确提示。

v1 不做自研播放器内核，不做转码，不做字幕增强能力。

### 房主推流模块

房主推流模块负责把房主本地播放的视频推给观众。

核心能力：

- 房主选择本地视频文件，文件不上传后端。
- 房主本地 HTML5 video 播放视频。
- 使用 `captureStream()` 获取播放器媒体流。
- 使用 WebRTC P2P mesh 给每个观众建立连接。
- 使用 WebSocket 转发推流信令。
- 观众端播放远端 MediaStream。
- 观众控制请求转发给房主端自动执行。

限制：

- 只承诺 Chrome/Edge 等支持 `captureStream()` 的现代浏览器。
- 本地文件刷新后需要重新选择。
- 不做独立字幕传输，字幕如需展示由房主本地播放器画面承载。
- 房主离开或断线时，房主推流房间直接关闭。

### 实时同步模块

实时同步模块使用 WebSocket。

核心能力：

- 房间成员建立 WebSocket 连接。
- 用户加入房间后订阅该房间消息。
- 播放控制事件从客户端发送到服务端。
- 服务端记录最新权威状态。
- 服务端广播状态给房间内其他成员。
- 新成员加入时立即下发当前完整房间状态。

### 语音模块

语音模块使用 WebRTC。

核心能力：

- 请求浏览器麦克风权限。
- 加入房间语音。
- 退出房间语音。
- 静音和取消静音。
- 使用 WebSocket 转发 WebRTC 信令。
- 使用 STUN 辅助公网连接。
- 在复杂网络环境下支持配置 TURN。

v1 推荐使用 P2P mesh 语音模式，部署简单，适合小房间。

## AList/OpenList 对接方式

后端统一封装 AList/OpenList 访问，不建议前端直接调用 AList/OpenList。

原因：

- 避免 AList/OpenList Token 暴露到浏览器。
- 统一处理鉴权、路径、错误和文件类型过滤。
- 后续可以增加目录白名单、访问限制和视频 URL 代理能力。

建议配置项：

- `ALIST_BASE_URL`：AList/OpenList 服务地址。
- `ALIST_TOKEN`：访问令牌。
- `ALIST_USERNAME`：可选，使用账号密码换取 Token 时使用。
- `ALIST_PASSWORD`：可选，使用账号密码换取 Token 时使用。
- `ALLOWED_ROOT_PATHS`：可选，允许访问的根目录白名单。

建议后端封装能力：

- 获取目录列表。
- 获取文件详情。
- 获取视频直链。
- 判断文件是否可能为视频资源。
- 将 AList/OpenList 错误转换为前端可理解的错误信息。

## 房间模型

房间状态建议使用内存结构起步，后续多实例部署时迁移到 Redis。

房间核心字段：

- `roomId`：房间唯一标识。
- `ownerId`：房主用户标识。
- `members`：当前成员列表。
- `currentVideo`：当前视频信息。
- `playbackState`：当前播放状态。
- `createdAt`：创建时间。
- `updatedAt`：最后更新时间。
- `emptySince`：房间变为空的时间。

当前视频信息：

- `filePath`：AList/OpenList 文件路径。
- `fileName`：文件名。
- `playUrl`：可播放地址。
- `mimeType`：媒体类型，可选。
- `size`：文件大小，可选。

播放状态：

- `playing`：是否正在播放。
- `positionSeconds`：服务端记录状态时的视频位置。
- `playbackRate`：播放倍速。
- `stateUpdatedAt`：状态更新时间。
- `version`：状态版本号，用于忽略旧消息。

## 播放同步机制

播放同步以服务端记录的房间状态为权威状态。

### 基础流程

1. 房主选择视频，客户端发送 `load_video`。
2. 服务端更新房间当前视频，重置播放位置并广播。
3. 房主点击播放，客户端发送 `play`，包含当前播放位置。
4. 服务端记录播放状态和更新时间，并广播给其他成员。
5. 成员客户端收到状态后加载视频并调整本地播放器。
6. 房主暂停或拖动进度时重复上述同步流程。

### 新成员加入同步

新成员加入房间时，服务端返回当前完整房间状态。

如果房间正在播放，客户端需要根据服务端状态更新时间计算目标播放位置：

目标位置 = 服务端记录位置 + 当前时间与状态更新时间的差值 × 播放倍速

如果房间处于暂停状态，则直接跳转到服务端记录位置并保持暂停。

### 漂移校准

为了避免不同客户端长时间播放后逐渐不同步，客户端需要定期校准。

建议策略：

- 每 5 到 10 秒请求或接收一次当前权威状态。
- 本地时间差小于 0.5 秒时不处理。
- 本地时间差在 0.5 到 1 秒之间时可轻微调整播放速度追赶。
- 本地时间差超过 1 秒时直接 seek 到目标位置。

### 控制权限

v1 默认只有房主可以控制播放。

可选扩展：

- 房主开启“允许所有人控制”。
- 房主转让控制权。
- 指定某个成员为临时控制者。

## 语音方案

语音使用 WebRTC 音频通话，后端不转发音频流，只转发信令。

### 推荐模式

v1 使用 P2P mesh：

- 每个成员与其他成员分别建立 WebRTC 连接。
- 后端通过 WebSocket 转发 offer、answer 和 ICE candidate。
- 浏览器之间直接传输音频。
- 部署成本低，不需要独立音视频服务器。

该模式适合小房间。人数增加时，每个客户端的连接数和上行带宽会明显增加。

### 信令消息

建议复用房间 WebSocket 连接传递信令：

- `voice_join`：加入语音。
- `voice_leave`：退出语音。
- `voice_offer`：发送 WebRTC offer。
- `voice_answer`：发送 WebRTC answer。
- `voice_ice_candidate`：发送 ICE candidate。
- `voice_mute`：同步静音状态，可选。

### STUN/TURN

建议默认配置公共或自建 STUN 服务。

如果用户处于复杂 NAT、公司网络或运营商限制环境，P2P 可能失败，此时需要配置 TURN 服务中继音频。

配置项建议：

- `WEBRTC_STUN_URLS`
- `WEBRTC_TURN_URLS`
- `WEBRTC_TURN_USERNAME`
- `WEBRTC_TURN_PASSWORD`

### 权限和体验

- 用户主动点击加入语音后再请求麦克风权限。
- 拒绝麦克风权限时，不影响视频播放和同步。
- 提供静音按钮。
- 提供成员语音状态展示。
- 用户退出房间时自动断开语音连接。

## 推荐技术栈

### 前端

推荐使用 Vue 3 或 React。

核心依赖：

- 前端框架：Vue 3 或 React。
- 构建工具：Vite。
- 播放器：HTML5 video。
- 实时连接：原生 WebSocket 或轻量封装。
- 语音：浏览器 WebRTC API。

### 后端

推荐使用 Spring Boot 3 和 JDK 17。

核心能力：

- REST API：提供选片、房间查询和创建接口。
- WebSocket：提供播放同步和 WebRTC 信令。
- AList/OpenList Client：封装网盘资源访问。
- Room Manager：维护房间和成员状态。

### 存储

v1 不需要数据库。

房间状态使用内存 Map：

- 开发和单机部署简单。
- 房间生命周期短。
- 无历史数据要求。

后续如需多实例部署，可以使用 Redis 存储房间状态，并使用 Redis Pub/Sub 或消息队列广播跨实例事件。

## 部署形态

### 单机部署

适合 v1：

- 一个前端静态站点。
- 一个后端服务。
- 一个已部署的 AList/OpenList 服务。
- 可选一个 TURN 服务。

单机部署链路简单，适合家庭服务器、NAS、小型 VPS。

### 反向代理

建议使用 Nginx 或 Caddy 做反向代理：

- 前端静态资源。
- 后端 REST API。
- WebSocket 升级。
- HTTPS 证书。

WebRTC 获取麦克风通常要求安全上下文，公网访问时建议启用 HTTPS。

### 多实例扩展

当需要多实例部署时，需要调整：

- 房间状态从内存迁移到 Redis。
- WebSocket 消息通过 Redis Pub/Sub 或消息队列跨实例广播。
- 语音如果人数增多，需要从 P2P mesh 改为 SFU。

- 不做资源上传。
